data-handlers/serve

Servidor HTTP completo para Node.js puro — roteamento estilo Bun, WebSocket nativo, rate limiting por IP, autenticação integrada, logger colorido e graceful shutdown. Tudo sem instalar mais nada.

Node.js ≥ 18 obrigatório. O serve usa Web APIs nativas (Request, Response, URL) que só existem a partir do Node 18. Se você já usa Bun, Express ou Fastify no projeto — não precisa disso.

Imports disponíveis #

Tudo que o subpath export data-handlers/serve expõe. Você só importa o que vai usar.

Tudo que pode ser importado de data-handlers/serve

Primeiro servidor #

O mínimo para ter um servidor funcionando é passar um objeto com routes. Tudo o mais é opcional. Veja abaixo três exemplos progressivos — do mais simples ao mais completo.

Exemplo 1 — o mínimo absoluto

Um servidor que responde { ok: true } em qualquer rota. Útil para testar se a instalação funcionou ou para servir um health check simples.

server.ts — o mínimo para rodar 
import { serve } from 'data-handlers/serve'

serve({
  port: 3000,
  routes: {
    '/': () => Response.json({ ok: true }),
  }
})

Pronto. Acesse http://localhost:3000/ e vai receber {"ok":true}. Se você não passar port, o padrão é 3000.

Exemplo 2 — API REST com múltiplas rotas

Um servidor real com rotas GET e POST, parâmetro dinâmico :id, logger ligado e um banner customizado no terminal quando o servidor iniciar.

server.ts — API de usuários completa 
import { serve } from 'data-handlers/serve'

const users = [{ id: '1', name: 'Ana' }, { id: '2', name: 'Carlos' }]

serve({
  port: 3000,
  logger: true,  // loga método, path, status e tempo de cada request

  routes: {
    // GET /users → lista todos
    '/users': {
      GET:  () => Response.json(users),
      POST: async (req) => {
        const body = await req.json()
        users.push(body)
        return Response.json(body, { status: 201 })
      },
    },

    // GET /users/1 → busca por id no req.params
    '/users/:id': {
      GET: (req) => {
        const user = users.find(u => u.id === req.params.id)
        return user
          ? Response.json(user)
          : Response.json({ error: 'not found' }, { status: 404 })
      },
    },
  },
}).listen({
  banner: (url) => console.log(`🚀 rodando em ${url}`),
})

Exemplo 3 — servidor de produção completo

Auth via JWT, rate limit, middleware global de CORS, rotas protegidas, WebSocket e graceful shutdown. Esse é o padrão para um servidor pronto para produção.

server.ts — configuração de produção 
import { serve, requireAuth, systemResponses } from 'data-handlers/serve'

serve({
  port: 3000,
  logger: { level: 'standard', ignore: ['/health'] },

  // auth roda em toda request — resultado vai para req.user
  auth: async (req) => {
    const token = req.headers.get('authorization')?.replace('Bearer ', '')
    if (!token) return null
    try { return await verifyJwt(token) } catch { return null }
  },

  // rate limit por IP — 100 requests por minuto
  rateLimit: { windowMs: 60_000, max: 100 },

  routes: {
    '/health': () => Response.json({ status: 'ok' }),

    // rota pública — qualquer um acessa
    '/users': {
      GET: () => Response.json(db.users.list()),
    },

    // rota protegida — requireAuth bloqueia com 401 se não autenticado
    '/me': {
      middleware: [requireAuth],
      GET: (req) => Response.json(req.user),
    },

    // WebSocket — upgrade acontece no handler GET
    '/ws': {
      middleware: [requireAuth],
      GET: (req) => req.upgrade({ user: req.user }),
    },
  },

  websocket: {
    open:    (ws) => ws.send(`bem-vindo, ${ws.data.user.name}`),
    message: (ws, data) => ws.send(`eco: ${data}`),
    close:   (ws, code) => console.log('desconectou', code),
  },

  shutdown: {
    timeout: 10_000,
    onShutdown: async () => {
      await db.close()
      console.log('banco fechado')
    },
  },

  error: (err) => Response.json({ error: String(err) }, { status: 500 }),
  fetch: (req) => systemResponses[404](req),

}).listen({
  banner: (url) => console.log(`🚀 ${url}`),
})

ServeOptions — referência completa #

Todas as opções passadas para serve(). Nenhuma é obrigatória — você começa com o que precisa e adiciona o resto conforme a aplicação cresce.

Opção Tipo Default Descrição
port number 3000 Porta em que o servidor vai escutar
host string '0.0.0.0' Interface de rede. Use '127.0.0.1' para expor só localmente
routes Routes {} Mapa de paths para handlers. Chave é o path, valor é um handler ou MethodMap
middleware Middleware[] [] Middlewares globais — rodam em toda request, em ordem, antes de qualquer handler de rota
fetch RouteHandler — Fallback quando nenhuma rota casa. Se omitido, usa systemResponses[404] automaticamente
error (err) => Response — Handler global para exceções não tratadas nos handlers. Se omitido, usa systemResponses[500]
auth AuthFunction<TUser> — Roda antes de todo middleware. Resultado injetado em req.user — nunca undefined
rateLimit RateLimitOptions — Rate limit por IP com sliding window. Headers automáticos em todas as respostas
logger boolean | LoggerOptions — true usa nível 'standard'. Objeto permite configuração completa
websocket WSHandlers — Handlers de WebSocket na mesma porta do HTTP. Upgrade feito via req.upgrade()
shutdown ShutdownOptions — Configura graceful shutdown automático no SIGINT/SIGTERM

Rotas #

Cada chave do objeto routes é um path. O valor pode ser um handler direto — que responde a qualquer método HTTP — ou um MethodMap: um objeto onde cada chave é um método HTTP (GET, POST, etc.) com seu próprio handler.

Rotas estáticas têm prioridade automática sobre dinâmicas quando os paths são ambíguos. Por exemplo, '/users/me' sempre vai ser preferida em relação a '/users/:id' quando o path for exatamente /users/me.

Todos os formatos de rota suportados 
serve({
  routes: {
    // Handler direto — responde independente do método HTTP
    '/': (req) => Response.json({ method: req.method }),

    // MethodMap — métodos HTTP separados na mesma rota
    '/users': {
      GET:    () => Response.json(users),
      POST:   async (req) => {
        const body = await req.json()
        /* ... */
      },
      PUT:    (req) => /* ... */,
      PATCH:  (req) => /* ... */,
      DELETE: (req) => /* ... */,
    },

    // Parâmetro dinâmico — req.params.id é a parte variável do path
    '/users/:id': {
      GET:    (req) => Response.json({ id: req.params.id }),
      DELETE: (req) => /* ... */,
    },

    // Múltiplos parâmetros — todos disponíveis em req.params
    '/users/:userId/posts/:postId': {
      GET: (req) => Response.json(req.params),
      // req.params → { userId: '42', postId: '7' }
    },

    // Wildcard — captura qualquer sub-path abaixo de /static/
    '/static/*': (req) => Response.json({ path: req.url }),
  },

  // Fallback — roda quando nenhuma rota casou
  fetch: (req) => Response.json({ error: 'not found' }, { status: 404 }),
})

O objeto req — o que está disponível

O req nos handlers é uma Request da Web API padrão, com três propriedades extras adicionadas pelo serve: params, user e o método upgrade().

BunLikeRequest — tudo que você pode usar no handler TypeScript 
import type { RouteHandler } from 'data-handlers/serve'

const handler: RouteHandler = async (req) => {
  // Extras do serve
  req.params          // { id: string } — parâmetros dinâmicos da rota
  req.user            // TUser | null — resultado da função auth
  req.upgrade(data)   // faz upgrade para WebSocket; data vai para ws.data

  // Web APIs padrão — funcionam igual ao browser/Bun
  const body    = await req.json()           // JSON → objeto
  const texto   = await req.text()           // string pura
  const form    = await req.formData()       // FormData (multipart)
  const binario = await req.arrayBuffer()    // binário

  // Leitura de headers e query string
  const auth = req.headers.get('authorization')
  const page = new URL(req.url).searchParams.get('page')

  return Response.json({ ok: true })
}
Limite de payload: o body é limitado a 1 MB por padrão. Payloads maiores encerram a conexão com erro — o handler error captura e pode retornar um 400 ou 413 customizado.

Quando uma rota existe mas o método HTTP solicitado não está definido no MethodMap, o servidor responde automaticamente com 405 Method Not Allowed. Quando nenhuma rota casa, cai no fetch customizado ou no systemResponses[404].

Middleware #

Middlewares são funções que rodam antes do handler de rota. O contrato é simples: retornar null passa para o próximo middleware (ou para o handler). Retornar uma Response interrompe a chain e responde imediatamente — o handler nunca chega a rodar.

A execução é sempre serial e na ordem declarada — nunca paralela. Middlewares globais rodam primeiro, depois os middlewares da rota específica, depois o handler.

Contrato de tipo

Interface Middleware TypeScript 
type Middleware<TUser> = (
  req: BunLikeRequest<TUser>
) => Response | null | Promise<Response | null>

// null     → continua para o próximo middleware ou handler
// Response → interrompe aqui e responde; nada mais roda

Exemplo 1 — middleware global de CORS

Um middleware que adiciona os headers de CORS em toda response. Nunca bloqueia — sempre retorna null. Este é o padrão para middlewares que só observam ou modificam headers.

Middleware global — CORS em todas as rotas 
import { serve } from 'data-handlers/serve'
import type { Middleware } from 'data-handlers/serve'

const cors: Middleware = (req) => {
  // Preflight — OPTIONS retorna imediatamente os headers CORS
  if (req.method === 'OPTIONS') {
    return new Response(null, {
      status: 204,
      headers: {
        'Access-Control-Allow-Origin':  '*',
        'Access-Control-Allow-Methods': 'GET, POST, PUT, PATCH, DELETE',
        'Access-Control-Allow-Headers': 'Content-Type, Authorization',
      },
    })
  }
  return null  // outros métodos → continua normalmente
}

serve({
  middleware: [cors],  // global — roda antes de toda rota
  routes: {
    '/api/users': { GET: () => Response.json([]) },
  },
})

Exemplo 2 — middleware de rota para verificar role

Um middleware que bloqueia usuários sem a role correta. Só é aplicado na rota onde for declarado — rotas públicas não são afetadas.

Middleware por rota — bloqueio por role de usuário 
import { serve, requireAuth, systemResponses } from 'data-handlers/serve'

// requireAuth → bloqueia com 401 se req.user for null
// onlyAdmin  → bloqueia com 403 se o usuário não for admin
function onlyAdmin(req) {
  if (req.user?.role !== 'admin')
    return systemResponses[403](req, 'Requer role admin')
  return null
}

serve({
  auth: async (req) => verifyToken(req.headers.get('authorization')),

  routes: {
    // pública — qualquer um acessa
    '/produtos': {
      GET: () => Response.json(db.produtos.list()),
    },

    // protegida — só usuário autenticado
    '/minha-conta': {
      middleware: [requireAuth],
      GET: (req) => Response.json(req.user),
    },

    // restrita — autenticado + admin. Cadeia: requireAuth → onlyAdmin → handler
    '/admin/usuarios': {
      middleware: [requireAuth, onlyAdmin],
      GET:    () => Response.json(db.users.list()),
      DELETE: (req) => Response.json(db.users.delete(req.params.id)),
    },
  },
})

Exemplo 3 — middleware de API key para webhooks

Padrão comum para proteger endpoints de webhook que recebem callbacks externos (Stripe, GitHub, etc.). A API key fica no header, verificada em tempo constante com safeCompare para evitar timing attacks.

Middleware — verificação de API key em tempo constante 
import { serve, safeCompare, systemResponses } from 'data-handlers/serve'

// safeCompare evita timing attacks — nunca use === para comparar tokens
function apiKey(req) {
  const key = req.headers.get('x-api-key') ?? ''
  if (!safeCompare(key, process.env.WEBHOOK_SECRET))
    return systemResponses[401](req, 'API key inválida')
  return null
}

serve({
  routes: {
    '/webhooks/stripe': {
      middleware: [apiKey],
      POST: async (req) => {
        const event = await req.json()
        // processa evento do Stripe com segurança
        return Response.json({ received: true })
      },
    },
  },
})
Ordem de execução: [middlewares globais] → [middlewares de rota] → handler. Se qualquer middleware retornar uma Response, os seguintes são ignorados e o handler nunca roda.

requireAuth — middleware pronto para uso

requireAuth — equivalente interno 
// O que requireAuth faz internamente — exatamente isso:
function requireAuth(req) {
  if (!req.user) return systemResponses[401](req)
  return null
}

// Uso simples — adiciona na array middleware da rota
'/perfil': {
  middleware: [requireAuth],
  GET: (req) => Response.json(req.user),  // req.user nunca é null aqui
}

Auth embutido #

O serve não impõe um mecanismo de autenticação — você define a lógica, ele cuida da integração. A função auth é chamada antes de qualquer middleware em toda request. O resultado é injetado em req.user e fica disponível em todos os middlewares e handlers.

req.user está sempre presente — nunca undefined. Quando auth retorna null, o usuário não está autenticado e req.user é null. Cabe ao middleware ou handler decidir o que fazer com isso.

Exemplo 1 — JWT com TypeScript genérico

Passe o tipo do usuário como parâmetro genérico no serve<TUser>() e o TypeScript vai inferir req.user como TUser | null em todos os handlers — sem nenhum cast manual.

Auth com JWT e tipagem completa 
import { serve, requireAuth } from 'data-handlers/serve'

interface User {
  id:   string
  name: string
  role: 'admin' | 'user'
}

serve<User>({
  auth: async (req): Promise<User | null> => {
    const token = req.headers.get('authorization')?.replace('Bearer ', '')
    if (!token) return null
    try {
      return await verifyJwt<User>(token)  // retorna User ou lança
    } catch {
      return null  // token expirado, malformado, etc.
    }
  },

  routes: {
    '/me': {
      middleware: [requireAuth],
      GET: (req) => {
        // req.user é User (não null) — requireAuth já garantiu
        return Response.json({ id: req.user!.id, role: req.user!.role })
      },
    },

    '/dados-sensiveis': {
      middleware: [requireAuth],
      GET: (req) => {
        if (req.user!.role !== 'admin')
          return Response.json({ error: 'acesso negado' }, { status: 403 })
        return Response.json(db.secret.get())
      },
    },
  },
})

Exemplo 2 — API key simples (sem JWT)

Para APIs internas ou scripts, JWT pode ser excessivo. Basta verificar um header customizado e retornar um objeto estático como "usuário".

Auth por API key — sem JWT 
import { serve, requireAuth, safeCompare } from 'data-handlers/serve'

serve({
  auth: async (req) => {
    const key = req.headers.get('x-api-key') ?? ''
    // safeCompare evita timing attacks na comparação
    if (!safeCompare(key, process.env.API_KEY)) return null
    return { id: 'service', role: 'admin' }  // usuário fixo
  },
  routes: {
    '/api/dados': {
      middleware: [requireAuth],  // bloqueia sem a key
      GET: () => Response.json(db.tudo()),
    },
  },
})

Rate limit #

Rate limit por IP usando algoritmo de sliding window — imune a burst attacks na borda da janela, diferente do algoritmo de fixed window que permite o dobro de requests em dois momentos consecutivos exatamente no limite da janela.

Estado armazenado em memória — é resetado ao reiniciar o processo. Em ambientes com múltiplas instâncias do servidor, use um store externo como Redis para compartilhar o estado entre processos.

Configuração completa do rate limit 
serve({
  rateLimit: {
    windowMs:   60_000,                                     // janela de 1 minuto em ms
    max:        100,                                        // máximo de requests por IP
    message:    'Limite atingido. Tente novamente em breve.', // mensagem no 429
    trustProxy: false,                                      // padrão — usa socket.remoteAddress
  },
})

O servidor adiciona automaticamente os headers de rate limit em todas as respostas, não apenas no 429 — permitindo que o cliente monitore o consumo restante a cada request.

Headers adicionados em todas as respostas 
X-RateLimit-Limit:     100     // max configurado
X-RateLimit-Remaining: 73      // requests restantes na janela atual
X-RateLimit-Reset:     1774…   // timestamp Unix do próximo reset
Retry-After:           42      // só no 429 — segundos até poder tentar de novo

Exemplo — rate limits diferentes por rota

O rate limit global protege todo o servidor. Se precisar de limites diferentes por contexto, combine um middleware customizado por rota com a opção global mais permissiva.

Rate limit mais restritivo no endpoint de login 
import { serve, systemResponses } from 'data-handlers/serve'

// Contador simples em memória para o endpoint de login
const loginAttempts = new Map<string, { count: number; reset: number }>()

function loginRateLimit(req) {
  const ip  = req.headers.get('x-forwarded-for') ?? 'unknown'
  const now = Date.now()
  const entry = loginAttempts.get(ip)

  if (!entry || entry.reset < now) {
    loginAttempts.set(ip, { count: 1, reset: now + 900_000 })  // janela de 15min
    return null
  }

  if (entry.count >= 5) {
    return systemResponses[429](req, 'Muitas tentativas. Aguarde 15 minutos.')
  }

  entry.count++
  return null
}

serve({
  rateLimit: { windowMs: 60_000, max: 200 },  // global — permissivo

  routes: {
    '/auth/login': {
      middleware: [loginRateLimit],  // restritivo — 5 tentativas por 15min
      POST: async (req) => {
        const { email, password } = await req.json()
        /* lógica de login */
      },
    },
  },
})
trustProxy: true usa o header X-Forwarded-For para extrair o IP real do cliente. Esse header é manipulável — qualquer cliente pode enviar um valor arbitrário. Só ative se você confia completamente no proxy ou load balancer na frente do servidor.

Logger #

Logger colorido com três níveis de detalhe. Roda após cada resposta — nunca bloqueia a request. Pode ser substituído completamente por um logger customizado via onLog.

Todos os modos de configurar o logger 
serve({
  // Modo simples — habilita com nível 'standard'
  logger: true,

  // Ou objeto completo com todas as opções
  logger: {
    level:  'verbose',           // 'minimal' | 'standard' | 'verbose'
    prefix: 'api',              // prefixo em cada linha: [api]
    ignore: ['/health', '/metrics', /\/ping/],  // rotas silenciadas (string ou regex)

    // onLog substitui o logger padrão completamente
    onLog: (entry) => {
      myMonitoring.track({
        method:   entry.method,
        path:     entry.pathname,
        status:   entry.status,
        duration: entry.durationMs,
        ip:       entry.ip,
      })
    },
  },
})
Nível O que é registrado
minimal Método, path, status e duração em ms
standard + headers da request relevantes (content-type, authorization, user-agent…)
verbose + body da response (se for JSON)

Exemplo — integração com Datadog / Sentry

Use onLog para enviar métricas para qualquer sistema de monitoramento. O campo status permite filtrar apenas os erros e o durationMs serve para detectar rotas lentas.

onLog — enviando métricas para monitoramento 
serve({
  logger: {
    level: 'standard',
    ignore: ['/health', '/favicon.ico'],

    onLog: (entry) => {
      // Sempre loga no console (substitua pelo seu logger favorito)
      console.log(`${entry.method} ${entry.pathname} → ${entry.status} (${entry.durationMs}ms)`)

      // Erros 5xx → rastreia no Sentry
      if (entry.status >= 500) {
        Sentry.captureMessage(`${entry.method} ${entry.pathname} → ${entry.status}`)
      }

      // Toda request → métricas no Datadog
      datadog.increment('http.requests', 1, [
        `method:${entry.method}`,
        `path:${entry.pathname}`,
        `status:${entry.status}`,
      ])
      datadog.histogram('http.duration_ms', entry.durationMs)
    },
  },
})

LogEntry — todos os campos disponíveis no onLog

Tipo LogEntry TypeScript 
type LogEntry = {
  method:          string                          // 'GET', 'POST'…
  pathname:        string                          // '/api/users'
  status:          number                          // 200, 404…
  durationMs:      number                          // tempo de processamento em ms
  requestHeaders:  Record<string, string>          // todos os headers da request
  responseBody:    string | null                  // body da response (só no verbose)
  timestamp:       string                          // ISO 8601 — quando a request chegou
  ip:              string                          // IP do cliente
}

WebSocket #

WebSocket nativo sem libs externas, na mesma porta do servidor HTTP. O upgrade é feito via req.upgrade(data) dentro de um handler GET — o objeto data que você passar fica disponível como ws.data em todos os handlers de WebSocket. Use isso para passar informações do contexto da request (usuário autenticado, sala, etc.).

Exemplo 1 — echo server simples

O mais básico: aceita qualquer conexão e devolve tudo que receber. Ótimo para entender o fluxo antes de adicionar autenticação.

Echo server — WebSocket mínimo 
import { serve } from 'data-handlers/serve'

serve({
  routes: {
    // req.upgrade() faz o handshake WebSocket
    // {} é o dado inicial — disponível em ws.data
    '/ws': { GET: (req) => req.upgrade({}) },
  },

  websocket: {
    open:    (ws)       => ws.send('conectado!'),
    message: (ws, data) => ws.send(`eco: ${data}`),
    close:   (ws, code) => console.log('desconectou, código:', code),
    error:   (ws, err)  => console.error('erro WS:', err),
  },
}).listen({ banner: (url) => console.log(`WS disponível em ${url}/ws`) })

Exemplo 2 — chat com autenticação e salas

Padrão mais realista: autentica o usuário antes do upgrade, passa o contexto para ws.data e usa isso nos handlers de mensagem para saber quem está falando.

WebSocket com auth — usuário disponível nos handlers 
import { serve, requireAuth } from 'data-handlers/serve'

// Rastreia as conexões ativas por sala
const salas = new Map<string, Set<any>>()

serve({
  auth: async (req) => verifyToken(req.headers.get('authorization')),

  routes: {
    '/ws/chat': {
      middleware: [requireAuth],  // bloqueia com 401 se não autenticado

      GET: (req) => {
        const sala = new URL(req.url).searchParams.get('sala') ?? 'geral'
        // tudo passado aqui fica em ws.data nos handlers abaixo
        return req.upgrade({ user: req.user, sala })
      },
    },
  },

  websocket: {
    allowedOrigins: ['https://meusite.com'],  // rejeita conexões de outras origens
    maxPayload:     32 * 1024,               // mensagens maiores que 32KB fecham a conexão
    heartbeat:      30_000,                  // ping/pong automático a cada 30s

    open: (ws) => {
      const { user, sala } = ws.data
      if (!salas.has(sala)) salas.set(sala, new Set())
      salas.get(sala)!.add(ws)
      ws.send(`Bem-vindo à sala "${sala}", ${user.name}!`)
    },

    message: (ws, data) => {
      const { user, sala } = ws.data
      const msg = `[${user.name}] ${data}`
      // broadcast para todos na mesma sala
      salas.get(sala)?.forEach(conn => conn.send(msg))
    },

    close: (ws) => {
      const { sala } = ws.data
      salas.get(sala)?.delete(ws)
    },
  },
})

Exemplo 3 — notificações server-sent sem polling

Use WebSocket para empurrar notificações em tempo real para o cliente — sem ele precisar ficar consultando a API. O servidor envia quando algo acontece no banco ou em outro processo.

WebSocket — notificações server-push 
import { serve, requireAuth } from 'data-handlers/serve'

// Guarda todas as conexões ativas por userId
const conexoes = new Map<string, any>()

serve({
  auth: async (req) => verifyToken(req.headers.get('authorization')),

  routes: {
    '/ws/notificacoes': {
      middleware: [requireAuth],
      GET: (req) => req.upgrade({ userId: req.user!.id }),
    },

    // Endpoint interno para disparar notificação (ex: chamado por job ou evento)
    '/internal/notify/:userId': {
      POST: async (req) => {
        const ws = conexoes.get(req.params.userId)
        const payload = await req.json()

        if (ws) {
          ws.send(JSON.stringify(payload))
          return Response.json({ entregue: true })
        }
        return Response.json({ entregue: false, motivo: 'usuário offline' })
      },
    },
  },

  websocket: {
    open:  (ws) => conexoes.set(ws.data.userId, ws),
    close: (ws) => conexoes.delete(ws.data.userId),
  },
})
Browser não envia headers no WebSocket. O browser não permite passar Authorization na abertura de uma conexão WebSocket. Use query string para o token: ws://host/ws?token=abc e leia com new URL(req.url).searchParams.get('token') antes de chamar req.upgrade().

systemResponses #

Objeto com factories de resposta JSON padronizadas para os códigos de erro mais comuns. O servidor usa internamente — mas você pode usar nos seus próprios handlers para manter o formato de erro consistente em toda a aplicação.

Shape da resposta

Estrutura JSON retornada por qualquer systemResponse 
{
  statusCode: 404,
  error:      "Not Found",
  message:    "Usuário não encontrado",    // customizável
  path:       "/api/users/abc",           // extraído de req.url
  timestamp:  "2026-04-23T12:00:00.000Z"  // momento da resposta
}
Usando systemResponses nos handlers 
import { serve, systemResponses } from 'data-handlers/serve'

serve({
  routes: {
    '/usuarios/:id': {
      GET: async (req) => {
        const user = await db.users.find(req.params.id)
        if (!user)
          return systemResponses[404](req, 'Usuário não encontrado')
        return Response.json(user)
      },

      DELETE: async (req) => {
        if (!req.user)
          return systemResponses[401](req)

        if (req.user.role !== 'admin')
          return systemResponses[403](req, 'Apenas admins podem deletar usuários')

        await db.users.delete(req.params.id)
        return Response.json({ deleted: true })
      },
    },
  },
})
Código error Usado automaticamente por
400 Bad Request — (uso manual nos handlers)
401 Unauthorized requireAuth quando req.user é null
403 Forbidden — (uso manual nos handlers)
404 Not Found Nenhuma rota casou e fetch não definido
405 Method Not Allowed Rota existe mas método não está no MethodMap
429 Too Many Requests Rate limit esgotado para o IP
500 Internal Server Error Exceção não tratada e handler error não definido
503 Service Unavailable Request chegou durante o graceful shutdown
Sem informação vazada. As respostas automáticas (404, 405, 500) nunca expõem stack trace, lista de rotas ou detalhes internos — apenas o shape padronizado acima.

Tratamento de erros #

Se um handler lançar uma exceção não tratada, o serve captura e chama o handler error. Se error não for definido, usa systemResponses[500] automaticamente.

Error handler global com log e resposta customizada 
serve({
  error: (err) => {
    // Em desenvolvimento: expõe a mensagem do erro
    if (process.env.NODE_ENV !== 'production') {
      return Response.json(
        { error: (err as Error).message, stack: (err as Error).stack },
        { status: 500 }
      )
    }

    // Em produção: mensagem genérica + log no Sentry
    Sentry.captureException(err)
    return Response.json({ error: 'Erro interno do servidor' }, { status: 500 })
  },

  routes: {
    '/explode': () => { throw new Error('💥') },  // capturado pelo error handler
  },
})

Graceful Shutdown #

O graceful shutdown orquestra o encerramento seguro do servidor — para novas conexões, aguarda requests ativas terminarem, encerra WebSockets abertos e chama hooks de limpeza antes de matar o processo. Isso evita requests cortadas no meio e garante que o banco seja fechado corretamente.

O que acontece em ordem

Passo O que acontece
1 Para de aceitar novas conexões — server.close(). Novas requests recebem 503 com Retry-After: 30
2 Envia close frame 1001 Going Away para todos os WebSockets rastreados e limpa o Set interno
3 Verifica activeRequests a cada 50ms. Continua quando chegar a zero ou o timeout for excedido
4 Destrói todas as conexões TCP restantes rastreadas em openConnections
5 Chama onShutdown() — fecha banco, limpa cache, faz flush de logs

Exemplo 1 — configuração automática via sinais

Quando shutdown é configurado no serve(), os sinais SIGINT e SIGTERM são registrados automaticamente. Ctrl+C ou kill disparam o graceful shutdown sem nenhum código extra.

Graceful shutdown automático no SIGTERM 
const server = serve({
  routes: { '/': () => Response.json({ ok: true }) },

  shutdown: {
    timeout:    10_000,                     // aguarda até 10s pelas requests ativas
    signals:    ['SIGINT', 'SIGTERM'],       // padrão — Ctrl+C ou kill
    onShutdown: async () => {
      await db.close()                       // fecha conexões com o banco
      await cache.flush()                    // grava dados pendentes no cache
      await logger.drain()                   // flush de logs em buffer
      console.log('✓ limpeza concluída')
    },
  },
})

Exemplo 2 — shutdown manual em testes ou scripts

Você pode chamar gracefulShutdown() diretamente — útil em testes de integração, scripts de manutenção ou quando um evento da aplicação sinaliza que é hora de parar.

Chamada manual — útil em testes e scripts 
const server = serve({ routes: { '/': () => Response.json({ ok: true }) } })
server.listen()

// Depois de rodar seus testes:
await server.gracefulShutdown()

// Ou com opções que sobrescrevem as configuradas no serve()
await server.gracefulShutdown({
  timeout:    2_000,     // timeout menor para testes não travarem
  onShutdown: async () => { /* limpeza específica de teste */ },
})

// Para parar sem esperar pelas requests ativas (não recomendado em produção):
await server.stop()
503 durante o shutdown. Qualquer request que chegar enquanto o servidor está encerrando recebe imediatamente 503 Service Unavailable com Connection: close e Retry-After: 30. Load balancers e proxies reversos reconhecem esse padrão e param de rotear tráfego para essa instância.

listen() — opções de inicialização #

O serve() retorna um objeto de servidor que ainda não está escutando. Você chama .listen() para iniciá-lo. Isso permite configurar o servidor antes de expô-lo à rede.

Todas as formas de chamar listen() 
const server = serve({ routes: { '/': () => Response.json({ ok: true }) } })

// 1. Sem argumentos — usa port e host definidos no serve()
server.listen()

// 2. Só callback — compatibilidade com Node.js server.listen(cb)
server.listen(() => console.log('pronto'))

// 3. Objeto de opções completo
server.listen({
  port:   4000,
  host:   '127.0.0.1',   // expõe só localmente
  logger: { level: 'verbose' },  // sobrescreve o logger configurado no serve()

  // banner substitui o log padrão de inicialização
  banner: (url) => {
    console.log(`
  ┌─────────────────────────────────────┐
  │  🚀  Servidor iniciado              │
  │  ${url.padEnd(35)}│
  │  Ambiente: ${(process.env.NODE_ENV ?? 'dev').padEnd(24)}│
  └─────────────────────────────────────┘`)
  },

  // onReady — hook que roda após o servidor estar escutando
  onReady: (url) => {
    setupHealthCheck(url)
    notifyDeploymentSystem(url)
  },
})

Acesso programático após o listen

Propriedades e métodos do server após listen() 
server.url                           // string — ex: 'http://localhost:3000'
await server.gracefulShutdown()      // encerramento seguro
await server.stop()                  // para imediatamente, sem esperar requests

safeCompare #

Compara duas strings em tempo constante — independente de onde elas divergem. O operador === do JavaScript pode terminar mais rápido quando as strings diferem logo no começo, e isso cria uma janela de timing attack que permite adivinhar tokens caractere por caractere. safeCompare sempre leva o mesmo tempo, eliminando essa vulnerabilidade.

safeCompare — quando e como usar 
import { safeCompare } from 'data-handlers/serve'

// ❌ Evite para tokens e senhas — vulnerável a timing attack
if (token === process.env.SECRET) { /* ... */ }

// ✅ Use safeCompare — tempo constante sempre
if (safeCompare(token, process.env.SECRET)) { /* ... */ }

// Retorna boolean — false se uma das strings for undefined/null
safeCompare('abc', 'abc')   // true
safeCompare('abc', 'xyz')   // false
safeCompare('abc', null)    // false
Regra simples: sempre que comparar um valor vindo de uma request (header, query, body) com um segredo armazenado (API key, webhook secret, token), use safeCompare em vez de ===.

Cheatsheet #

data-handlers/serve — referência completa 
import { serve, requireAuth, systemResponses, safeCompare } from 'data-handlers/serve'

const server = serve({
  port:   3000,
  host:   '0.0.0.0',
  logger: { level: 'standard', ignore: ['/health'], onLog: (entry) => {} },

  auth: async (req) => verifyToken(req.headers.get('authorization')),

  rateLimit: { windowMs: 60_000, max: 100, trustProxy: false },

  middleware: [corsMiddleware],  // global

  routes: {
    '/':                () => Response.json({ ok: true }),
    '/users':           { GET: getUsers, POST: createUser },
    '/users/:id':       { GET: getUser, PATCH: updateUser, DELETE: deleteUser },
    '/me':              { middleware: [requireAuth], GET: (req) => Response.json(req.user) },
    '/ws':              { middleware: [requireAuth], GET: (req) => req.upgrade({ user: req.user }) },
    '/static/*':        (req) => serveStatic(req.url),
  },

  websocket: {
    allowedOrigins: ['https://meusite.com'],
    maxPayload:     64 * 1024,
    heartbeat:      30_000,
    open:    (ws)       => ws.send('bem-vindo'),
    message: (ws, data) => ws.send(`eco: ${data}`),
    close:   (ws, code) => console.log('fechou:', code),
    error:   (ws, err)  => console.error('erro:', err),
  },

  shutdown: {
    timeout:    10_000,
    signals:    ['SIGINT', 'SIGTERM'],
    onShutdown: async () => await db.close(),
  },

  error: (err) => Response.json({ error: String(err) }, { status: 500 }),
  fetch: (req) => systemResponses[404](req),
})

server.listen({
  banner:  (url) => console.log(`🚀 ${url}`),
  onReady: (url) => setupHealthCheck(url),
})

requireAuth                          // middleware: 401 se req.user === null
