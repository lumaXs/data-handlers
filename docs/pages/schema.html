Schemas

Quando você tem um payload com múltiplos campos, validar campo por campo com handlers individuais fica trabalhoso e propenso a esquecer alguma coisa. O sistema de schemas resolve isso — você define o shape do objeto uma vez e chama .parse() ou .safeParse().

Definindo um schema #

Importe schema e passe um objeto onde cada chave é um campo e o valor é o tipo (forma curta) ou um objeto de configuração (forma completa). Os tipos podem ser qualquer string registrada no registry — incluindo tipos de plugins.

Definindo um schema — forma curta 
import { schema } from 'data-handlers'

const cadastroSchema = schema({
  name:     'name',
  document: 'cpf',
  phone:    'phone',
  email:    'email',
})

parse() e safeParse() #

parse() processa o objeto e retorna todos os campos normalizados, ou lança um SchemaError com o mapa completo de erros se algum campo for inválido. safeParse() faz o mesmo, mas nunca lança — retorna { success, data, errors }.

Todos os erros de uma vez: o schema não para no primeiro campo inválido. Ele processa todos os campos antes de retornar — assim o usuário vê todos os problemas ao mesmo tempo, não um por um.
parse() — lança SchemaError se algum campo falhar 
const dados = cadastroSchema.parse({
  name:     'JOAO SILVA',
  document: '11144477735',
  phone:    '11987654321',
  email:    'JOAOsilVa95@Email.com',
})

// dados →
// {
//   name:     'Joao Silva',
//   document: '111.444.777-35',
//   phone:    '(11) 98765-4321',
//   email:    'joaosilva95@email.com'
// }
safeParse() — nunca lança, retorna o resultado 
const result = cadastroSchema.safeParse(req.body)

if (!result.success) {
  return res.status(400).json({ errors: result.errors })
  // result.errors →
  // {
  //   name:     '[normalize:name] Expected non-empty string...',
  //   document: '[normalize:cpf] Invalid CPF...',
  // }
}

await salvarUsuario(result.data) // já normalizado, pronto pro banco
Propriedade Quando success: true Quando success: false
success true false
data Objeto com todos os campos normalizados null
errors null Mapa { campo: mensagem } de todos os erros

Opções de campo #

Além da forma curta (só o tipo como string), cada campo aceita um objeto com configurações. As propriedades optional, default, options e label controlam como o campo é tratado durante o parse.

Todas as opções de campo em um schema 
const pedidoSchema = schema({
  customerName: 'name',

  coupon: {
    type:     'slug',
    optional: true,     // null/undefined passam sem erro
  },

  country: {
    type:    'slug',
    default: 'brasil', // usado quando o campo vier undefined
  },

  total: {
    type:    'number',
    options: { style: 'currency', currency: 'BRL', locale: 'pt-BR' },
  },

  doc: {
    type:  'cpf',
    label: 'CPF do cliente',  // aparece nas mensagens de erro
  },
})
Opção Tipo Descrição
type string Tipo registrado no registry (obrigatório na forma completa)
optional boolean Se true, null e undefined passam sem erro
default any Valor usado quando o campo vier undefined. Diferente de optional: o default é normalizado
options object Repassado ao handler como segundo argumento
label string Nome legível usado nas mensagens de erro em vez do nome da chave

Utilitários de schema #

Todos os métodos de utilitário retornam novos schemas — o schema original nunca é modificado. Isso permite derivar múltiplas variações a partir de um schema base sem efeitos colaterais.

partial · pick · omit · extend — todos imutáveis 
const userSchema = schema({ name: 'name', email: 'email', document: 'cpf' })

// .partial() — todos os campos viram opcionais
const updateSchema = userSchema.partial()
updateSchema.safeParse({})
// → { success: true, data: {}, errors: null }

// .pick() — novo schema só com os campos especificados
const loginSchema = userSchema.pick('email', 'document')

// .omit() — remove um ou mais campos
const publicUser = userSchema.omit('document')

// .extend() — adiciona campos (não altera o schema original)
const fullSchema = userSchema.extend({ phone: 'phone', cep: 'cep' })
Padrão prático — um schema base, múltiplas operações 
const userSchema = schema({ name: 'name', email: 'email', document: 'cpf', phone: 'phone' })

const createUser  = userSchema                   // POST — todos obrigatórios
const patchUser   = userSchema.partial()         // PATCH — todos opcionais
const publicUser  = userSchema.omit('document') // GET — sem CPF na resposta
const loginSchema = userSchema.pick('email')    // POST /login — só email

SchemaError #

Quando .parse() falha, lança uma instância de SchemaError com a propriedade .errors contendo o mapa completo de campos com erro. Você pode verificar com instanceof para distinguir de outros erros.

Capturando e inspecionando um SchemaError 
import { schema, SchemaError } from 'data-handlers'

try {
  cadastroSchema.parse({ name: '', document: 'invalido', phone: '123' })
} catch (err) {
  if (err instanceof SchemaError) {
    console.log(err.errors)
    // {
    //   name:     '[normalize:name] Expected non-empty string...',
    //   document: '[normalize:cpf] Invalid CPF. Received: invalido',
    //   phone:    '[normalize:phone] Expected 10 or 11-digit phone number'
    // }
  }
}
SchemaError em um endpoint Express com tratamento adequado 
import { schema, SchemaError } from 'data-handlers'

app.post('/usuarios', (req, res, next) => {
  try {
    const data = userSchema.parse(req.body)
    db.users.create(data)
    res.status(201).json(data)
  } catch (err) {
    if (err instanceof SchemaError) {
      return res.status(400).json({ message: 'Dados inválidos', errors: err.errors })
    }
    next(err) // outros erros vão pro handler de erro do Express
  }
})
parse() vs safeParse() em APIs: para endpoints onde você quer um único bloco de tratamento de erro, parse() com try/catch é mais limpo. Para handlers onde você quer retornar o erro imediatamente sem fluxo alternativo, safeParse() é mais direto. Ambos retornam o mesmo mapa de erros.
