data-handlers

Biblioteca de normalização e validação de dados para Node.js — sem dependências externas, com suporte nacional de primeira classe e um sistema de schemas para validar objetos inteiros de uma vez.

Handlers do core

  • name — Title Case + conectivos
  • number — Intl.NumberFormat
  • date — Intl.DateTimeFormat
  • password, url, uuid, any

Plugins nacionais

  • CPF, CNPJ, RG
  • CEP, Telefone
  • Slug, E-mail, Cor

API fluente

  • handlers.cpf.parse()
  • handlers.cpf.safe()
  • handlers.has() · .types

Schemas

  • Valida objetos inteiros
  • parse · safeParse
  • extend · pick · omit

Provavelmente você já escreveu este tipo de código em vários lugares do seu sistema — normalização e validação espalhadas, cada uma com sua implementação diferente, difíceis de manter e de testar:

Sem data-handlers — espalhado, inconsistente 
// Em algum lugar do projeto...
const nome = input.trim().toLowerCase().replace(/\s+/g, ' ')
  .split(' ').map(w => w[0].toUpperCase() + w.slice(1)).join(' ')

// Em outro lugar...
const cpf = raw.replace(/\D/g, '')
  .replace(/(\d{3})(\d{3})(\d{3})(\d{2})/, '$1.$2.$3-$4')
if (!cpfRegex.test(cpf)) throw new Error('CPF inválido')

// E em outro...
const valor = new Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'BRL' }).format(n)
Com data-handlers — uma API consistente para tudo 
import { handlers } from 'data-handlers'

handlers.name.normalize('  joao   silva  ')                                       // 'Joao Silva'
handlers.cpf.normalize('11144477735')                                            // '111.444.777-35'
handlers.number.normalize(1234.5, { style: 'currency', currency: 'BRL' })      // 'R$ 1.234,50'

Instalação #

Terminal 
npm install data-handlers
# ou
pnpm add data-handlers
Requisito: Node.js 18 ou superior. A lib usa ESM (type: "module"). O autocomplete TypeScript funciona em JS puro via os arquivos .d.ts inclusos — nenhuma configuração extra necessária.

Conceito central #

A lib mantém um registry interno — um Map que associa um nome de tipo a uma função chamada handler. Handlers do core são registrados automaticamente na importação. Plugins adicionam novos tipos ao mesmo registry.

Como o registry funciona internamente 
// O registry é um Map simples
registry.get('name')   // → nameHandler()
registry.get('number') // → numberHandler()
registry.get('cpf')    // → cpfHandler()  ← registrado pelo plugin

// Quando você chama handlers.name.normalize('joao'), internamente é:
const handler = registry.get('name')
return handler('joao')  // → 'Joao'

Um handler é qualquer função (value, options?) => string que retorna o valor normalizado ou lança um TypeError descritivo se o valor for inválido. A interface é intencionalmente simples — isso é o que torna a lib extensível sem cerimônia.

Padrão: endpoint REST #

O uso mais comum é combinar schemas com handlers para validar e normalizar o payload de uma request antes de salvar no banco. O schema processa todos os campos e retorna os erros de uma vez — sem precisar de múltiplas validações manuais.

Express — POST /usuarios 
import { schema, SchemaError } from 'data-handlers'

const cadastroSchema = schema({
  name:     'name',
  document: 'cpf',
  phone:    { type: 'phone', optional: true },
  email:    'email',
  zipCode:  { type: 'cep', label: 'CEP' },
})

app.post('/usuarios', (req, res) => {
  const result = cadastroSchema.safeParse(req.body)

  if (!result.success) {
    return res.status(400).json({
      message: 'Dados inválidos',
      errors:  result.errors,
    })
  }

  // result.data está completamente normalizado e pronto pro banco
  db.users.create(result.data)
  res.status(201).json(result.data)
})

Padrão: validação de formulário #

Handlers individuais são ideais para validação em tempo real de inputs — você usa .safe() para nunca travar o fluxo e exibir feedback imediato ao usuário enquanto ele digita.

Feedback em tempo real no input 
import { handlers } from 'data-handlers'

inputCPF.addEventListener('input', (e) => {
  const { valid, value, error } = handlers.cpf.safe(e.target.value)

  if (valid) {
    e.target.value = value          // aplica a máscara automaticamente
    campoErro.textContent = ''
    inputCPF.classList.remove('error')
  } else {
    campoErro.textContent = 'CPF inválido'
    inputCPF.classList.add('error')
  }
})

Padrão: formatar antes de exibir #

Dados geralmente chegam do banco em formato bruto — números sem máscara, datas como timestamp, nomes em maiúsculo. Handlers resolvem isso na camada de apresentação sem alterar os dados originais.

Formatando dados vindos do banco para exibição 
import { handlers } from 'data-handlers'

function formatarPerfil(usuario) {
  return {
    ...usuario,
    name:      handlers.name.normalize(usuario.name),
    document:  handlers.cpf.normalize(usuario.document),
    phone:     handlers.phone.normalize(usuario.phone),
    createdAt: handlers.date.normalize(usuario.createdAt, {
      dateStyle: 'long',
      locale:    'pt-BR',
    }),
    balance: handlers.number.normalize(usuario.balance, {
      style:    'currency',
      currency: 'BRL',
      locale:   'pt-BR',
    }),
  }
}

Padrão: PATCH parcial #

Ao usar .partial() em um schema, todos os campos se tornam opcionais — perfeito para endpoints PATCH onde o cliente envia apenas os campos que quer atualizar. O schema normaliza só o que chegou e ignora o resto.

Um schema base, duas operações diferentes 
const userSchema   = schema({ name: 'name', email: 'email', phone: 'phone' })
const updateSchema = userSchema.partial()  // todos os campos viram opcionais

app.patch('/usuarios/:id', (req, res) => {
  const result = updateSchema.safeParse(req.body)

  if (!result.success) {
    return res.status(400).json({ errors: result.errors })
  }

  // Só atualiza os campos que vieram — os outros ficam inalterados no banco
  db.users.update(req.params.id, result.data)
  res.json(result.data)
})
